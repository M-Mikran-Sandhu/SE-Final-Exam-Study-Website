Lecture 11: Requirements & Metrics

What is SRS? (Software Requirements Specification)

Simple Meaning: A document that lists everything the software should do.

Example: When you order food on Zomato, the SRS would say:

  • User can search for restaurants
  • User can view menu items
  • User can add items to cart
  • User can pay online

Verification vs Validation

Verification: "Are we building the product right?" (Check if code matches requirements)

Validation: "Are we building the right product?" (Check if it's what customer wants)

Example:

  • Verification: Code correctly implements the login feature
  • Validation: Customer actually needs a login feature

Requirements Elicitation (Gathering Requirements)

Simple Meaning: Asking customers what they want.

Methods:

  • Interviews: Talk to users directly
  • Surveys: Ask questions in a form
  • Observation: Watch how people work
  • Workshops: Group discussions

Metrics (How to Measure Software)

Measure: Raw data (e.g., 5000 lines of code)

Metric: Calculated value (e.g., 5000 lines / 10 developers = 500 lines per developer)

Indicator: What the metric tells us (e.g., productivity is high or low)

Example:

  • Measure: 50 bugs found
  • Metric: 50 bugs / 10,000 lines = 0.5 bugs per 100 lines
  • Indicator: Code quality is good

Function Points (FP)

Simple Meaning: Count how many features the software has.

Example: A banking app has:

  • Login (1 function)
  • View balance (1 function)
  • Transfer money (1 function)
  • Pay bills (1 function)
  • Total = 4 function points

Lecture 12: Metrics & Architecture Styles

Project Metrics vs Product Metrics

Project Metrics: How well the team is working (time, cost, effort)

Product Metrics: How good the software is (quality, speed, size)

Example:

  • Project: Project took 6 months, cost $100,000, 10 developers
  • Product: Software has 50,000 lines, 5 bugs, runs in 2 seconds

Lines of Code (LOC) vs Function Points (FP)

LOC: Count every line of code written

FP: Count features/functions

Example:

  • LOC: 10,000 lines of code
  • FP: 50 features
  • LOC depends on language (Python vs Java = different)
  • FP is the same no matter what language

Architecture Styles (How to Build Software)

1. Client-Server

Idea: One main computer (server) serves many users (clients)

Example: Gmail - Your browser is the client, Google's computer is the server

2. Peer-to-Peer (P2P)

Idea: All computers are equal, no main server

Example: BitTorrent - Everyone shares files with everyone

3. Layered Architecture

Idea: Software has layers - UI layer, Business layer, Database layer

Example: Instagram

  • UI Layer: What you see on screen
  • Business Layer: Logic for posting, liking, commenting
  • Database Layer: Stores all data

4. Publish-Subscribe

Idea: Publishers send messages, subscribers receive them (no direct connection)

Example: YouTube notifications - You subscribe to a channel, YouTube publishes new videos, you get notified

Lecture 13: Modularity & Design

Coupling (How Connected are Modules?)

Low Coupling (Good): Modules are independent, easy to change

High Coupling (Bad): Modules depend on each other, hard to change

Example:

  • Low Coupling: Login module doesn't need to know about Payment module
  • High Coupling: If you change Login, Payment breaks

Cohesion (Does Module Have One Purpose?)

High Cohesion (Good): Module does one thing well

Low Cohesion (Bad): Module does many unrelated things

Example:

  • High Cohesion: Login module only handles user login
  • Low Cohesion: Login module handles login, payment, and email

Cost-Benefit Analysis (Is It Worth It?)

ROI (Return on Investment): How much money you make back

Formula: (Profit - Cost) / Cost × 100%

Example:

  • Cost to build feature: $10,000
  • Profit from feature: $15,000
  • ROI = (15,000 - 10,000) / 10,000 × 100% = 50%
  • Meaning: You make 50% profit

Lecture 14: Quality & Estimation

Defect Removal Efficiency (DRE)

Simple Meaning: How many bugs you find before release

Formula: (Bugs found before release / Total bugs) × 100%

Example:

  • You find 90 bugs before release
  • Customers find 10 bugs after release
  • Total bugs = 100
  • DRE = 90/100 × 100% = 90% (Good!)

3-Point Estimation (Guess How Long It Takes)

Simple Meaning: Make 3 guesses - best case, worst case, most likely

Formula: (Best + 4×Most Likely + Worst) / 6

Example:

  • Best case: 5 days
  • Most likely: 10 days
  • Worst case: 20 days
  • Estimate = (5 + 4×10 + 20) / 6 = 65/6 ≈ 11 days

COCOMO Model (Estimate Project Cost)

Simple Meaning: Formula to guess how much effort a project needs

Effort = a × (Lines of Code)^b

Example:

  • Project has 10,000 lines of code
  • Using COCOMO: Effort = 2.4 × (10)^1.05 ≈ 25 person-months
  • Meaning: Need 25 months of work from 1 person (or 2.5 months from 10 people)

Lecture 15: Testing & Integration

Why Testing is Important

Simple Meaning: Check if software works before giving to customers

Benefits:

  • Find bugs early (cheaper to fix)
  • Make sure software does what it should
  • Customers are happy
  • Save money on fixing bugs later

Black Box vs White Box Testing

Black Box Testing

Idea: Test without knowing how code works

Example: Test login form

  • Try empty username
  • Try wrong password
  • Try correct login

White Box Testing

Idea: Test knowing how code works

Example: Test every line of login code

  • Test if statement works
  • Test database connection
  • Test password encryption

Levels of Testing

Unit Testing: Test one function/module alone

Integration Testing: Test how modules work together

System Testing: Test the whole software

Acceptance Testing: Customer tests if they like it

Example:

  • Unit: Test login function alone
  • Integration: Test login + database together
  • System: Test whole app (login, payment, profile)
  • Acceptance: Customer uses app and says "yes, I like it"

Cyclomatic Complexity (How Complex is Code?)

Simple Meaning: Count how many different paths code can take

Example:

  • Simple code (no if/else): 1 path
  • Code with 1 if: 2 paths
  • Code with 2 ifs: 4 paths
  • More paths = harder to test = more bugs

Integration Testing Approaches

Big Bang: Test all modules together at once

Top-Down: Test main module first, then smaller ones

Bottom-Up: Test small modules first, then combine

Sandwich: Mix of top-down and bottom-up
